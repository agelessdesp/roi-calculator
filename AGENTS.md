# ROI-калькулятор imot.io

## Назначение проекта

Проект представляет собой ROI-калькулятор для продукта **imot.io**. Калькулятор помогает оценить экономический эффект от внедрения речевой аналитики, сравнить сценарии окупаемости и подготовить понятное описание результата для клиента.

Интерфейс и контент проекта русскоязычные.

## Текущая структура проекта

- Основная логика находится в одном HTML-файле с inline CSS и inline JS.
- В каталоге также могут лежать резервные копии HTML-файла и служебная документация.
- Сборки, модулей и отдельного frontend/backend-разделения нет.

На практике это означает:
- разметка, стили и скрипты тесно связаны между собой;
- большинство изменений требуют аккуратной правки прямо внутри одного HTML-файла;
- любые крупные рефакторинги лучше делать поэтапно, чтобы не сломать связанные участки UI и расчётов.

## Технический стек

- Один HTML-файл как основная точка входа.
- Inline CSS.
- Inline JavaScript без модульной структуры.
- `Chart.js` через CDN для графика окупаемости.
- `html2canvas` через CDN для рендера DOM перед экспортом.
- `jsPDF` через CDN для генерации PDF.
- `Google Fonts` для шрифтов интерфейса.

## Архитектурная модель

Файл проекта организован по схеме:

1. HTML-разметка интерфейса.
2. Блок `<style>` со всеми стилями.
3. Блок `<script>` со всеми расчётами, глобальными объектами и обработчиками.

Особенности архитектуры:

- CSS использует короткие имена классов.
- JavaScript построен вокруг глобальных переменных и функций.
- Явного разделения на слои данных, представления и вычислений нет.
- Большая часть UI обновляется напрямую из функций расчёта.

Это монолитный одностраничный калькулятор. Любые изменения нужно проверять не только визуально, но и по связанным вычислениям, потому что многие блоки завязаны друг на друга через общее глобальное состояние.

## Основная функциональность

### Входные данные

Калькулятор принимает:

- данные клиента: обращения в месяц, конверсия, средний чек;
- дополнительные источники эффекта: упущенные продажи, допродажи, экономия на обучении;
- параметры тарифа;
- параметры сценария внедрения;
- отраслевые пресеты;
- параметры ramp-up модели.

### Тарифная часть

В проекте есть три режима работы с тарифом:

- `sel` — ручной выбор тарифа;
- `clc` — подбор рекомендуемого тарифа по объёму обращений и длительности звонка;
- `man` — ручной ввод стоимости лицензии и внедрения.

Для периодов оплаты используются скидки:

- `1` месяц: `0%`;
- `3` месяца: `6%`;
- `6` месяцев: `12%`;
- `12` месяцев: `25%`.

### Сценарное моделирование

Проект считает три сценария:

- консервативный;
- базовый;
- оптимистичный.

Для каждого сценария рассчитываются:

- ROI за год;
- ежемесячный чистый эффект;
- срок окупаемости;
- влияние ramp-up;
- показатели "до / после";
- вклад отдельных точек роста.

### Ramp-up модель

Ramp-up хранится по размеру бизнеса и сценарию. Первый месяц во всех сценариях считается нулевым по эффекту, так как это период настройки и адаптации.

Размер бизнеса определяется автоматически по количеству операторов:

- `< 15` — `small`;
- `15-49` — `medium`;
- `50+` — `large`.

### Генерация результата

Калькулятор умеет:

- показывать сравнительные метрики по ролям;
- строить график окупаемости;
- собирать текстовое описание результата;
- адаптировать вывод под выбранную роль;
- формировать PDF-версию отчёта.

### PDF-экспорт

Экспорт в PDF строится через отдельный скрытый DOM-блок `#pdfContent`, который рендерится в canvas, а затем собирается в PDF.

В PDF включаются:

- шапка и дата;
- входные параметры;
- метрики по роли;
- точки роста;
- сценарии;
- блоки сравнения "до / после";
- график;
- текстовое описание;
- дисклеймер.

Если контент не помещается на одну страницу A4, он разбивается на несколько страниц.

## Пользовательские роли

Поддерживаются четыре роли:

- `own` — собственник;
- `sal` — руководитель продаж / руководитель КЦ;
- `mar` — директор по маркетингу;
- `com` — коммерческий директор.

Роль влияет на:

- набор отображаемых метрик;
- формулировки в описании результата;
- акценты в персонализированном тексте;
- состав карточек и summary-блоков.

## Персонализация текста

Система генерации текста включает:

- несколько шаблонов описания результата;
- адаптацию под роль;
- адаптацию под выбранный сценарий;
- wizard персонализации;
- блок привязки клиентских болей к рассчитанным метрикам.

Важные сущности:

- боли клиента;
- уточняющие контексты;
- связки болей с метриками;
- контраргументы к уже опробованным подходам;
- завершающие фразы по ожиданиям клиента.

## Ключевые JS-объекты

Ниже перечислены основные глобальные объекты, на которых держится логика калькулятора:

- `T` — тарифы: названия, цены, лимиты минут.
- `D` — скидки по периоду оплаты.
- `P` — отраслевые пресеты.
- `RC` — конфигурация метрик по ролям.
- `RAMP` — ramp-up модели по размеру бизнеса и сценарию.
- `SCENARIOS` — описания сценариев, детали и вероятности.
- `PAINS` — список типовых болей клиента.
- `PAIN_CTX` — уточняющие вопросы и контекст.
- `PAIN_HOOKS` — текстовые шаблоны по болям.
- `PAIN_REL` — связь болей с метриками калькулятора.
- `PAIN_FILL` — автозаполнение дополнительных блоков по болям.
- `TRIED_COUNTER` — контраргументы на ранее пробованные решения.
- `EXPECT_CLOSE` — завершающие формулировки по ожиданиям.
- `R` — глобальное состояние рассчитанного результата.

## Ключевые функции

Основные функции, на которые стоит смотреть при изменениях:

- `calc()` — главный перерасчёт, обновление результатов и синхронизация UI.
- `genM()` — генерация текстового описания результата.
- `calcPaybackWithRamp(...)` — расчёт окупаемости с учётом ramp-up.
- `calcPaybackData(...)` — данные по месяцам для графика и сопутствующих блоков.
- `renCh()` — отрисовка графика через `Chart.js`.
- `updK()` и `buildKd()` — обновление карточек метрик по роли.
- `openWz()` и `renWz()` — wizard персонализации.
- `exportPDF()` — подготовка PDF и выгрузка.
- `setSc()`, `updScDetail()`, `hlScCol()` — управление сценариями.
- `updateRampTable()` — обновление таблицы ramp-up.
- `renPainGrowth()` — блок "Ваши точки роста".
- `calcT()` — расчёт рекомендованного тарифа.
- `renCR(rec)` — визуализация рекомендованного тарифа.
- `selRP(p)` — выбор периода оплаты в блоке рекомендации.
- `apRC()` — применение рекомендации к активному тарифному состоянию.
- `selT(t)` и `selP(p)` — выбор тарифа и периода оплаты.
- `swT(t)` — переключение вкладок тарифного режима.
- `autoDetectSize(ops)` — определение размера бизнеса по числу операторов.

## Глобальные переменные состояния

При правках важно учитывать основные глобальные переменные:

- `sT` — выбранный тариф.
- `sP` — выбранный период оплаты.
- `aT` — активная вкладка тарифа.
- `role` — текущая роль пользователя.
- `kSc` — выбранный сценарий.
- `bizSize` — размер бизнеса.
- `R` — итоговое состояние расчётов.
- `rT`, `rP`, `rRec` — промежуточное состояние тарифной рекомендации.

## Соглашения по коду

- Имена CSS-классов сокращены и не всегда самодокументируемы.
- Имена JS-функций тоже часто сокращены.
- Значения из полей обычно читаются через вспомогательные функции наподобие `gv(id)`.
- Перерасчёт запускается практически на каждом `input`-событии.
- PDF-стили обычно имеют префикс `.pdf-`.
- У тарифной логики есть отдельные служебные классы для переключателей и рекомендаций.

Перед переименованием любых сокращённых сущностей стоит проверить, не используются ли они сразу в нескольких несвязанных блоках.

## Правила внесения изменений

- Сначала определяйте, к какому блоку относится задача: расчёты, тарифы, роли, тексты, PDF или UI.
- Любые изменения расчётов проверяйте минимум в трёх сценариях.
- Любые изменения тарифной логики проверяйте во всех трёх режимах: `sel`, `clc`, `man`.
- Любые изменения текста проверяйте хотя бы на двух ролях.
- Любые изменения PDF проверяйте через реальный экспорт, а не только визуально в DOM.
- Не выносите код в отдельные модули без явной необходимости: проект пока поддерживается как единый HTML-файл.

## Практические замечания для работы с файлами

- В проекте используются файлы и имена с кириллицей.
- При чтении и записи файлов нужно явно держать `UTF-8`.
- Если инструмент плохо работает с кириллическими именами, лучше использовать PowerShell с явной кодировкой или точные пути, а не полагаться на неустойчивые маски.

## Что обычно ломается при неаккуратных правках

- Синхронизация между формой и расчётом.
- Привязка выбранного сценария к детализации и графику.
- Переключение тарифных вкладок и применение рекомендации.
- Содержимое PDF-версии, если меняется DOM или классы.
- Генерация текстов, если меняются входные поля, роли или структура `R`.

## Рекомендуемый порядок анализа перед правкой

1. Найти, какой HTML-блок отвечает за UI задачи.
2. Найти связанный обработчик или функцию перерасчёта.
3. Проверить, влияет ли изменение на `R`, сценарии, тарифы или PDF.
4. После правки вручную проверить интерфейс, сценарии и экспорт, если он затронут.
